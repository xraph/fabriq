# Phase 3 Backend Report — adminapi plugin-registry CRUD

## Persistence mechanism

**Chosen: dynamic entity (`admin_plugin_remote`) via the existing command plane.**

No SQL migration was written. The fake test harness (`fabriqtest.World`) handles
dynamic entities natively through `FakeStore`/`FakeRelational`; the real adapter
uses `EnsureDynamic`-managed DDL when connected to Postgres. This is identical to
the `widget` entity already used by the pre-existing adminapi tests and is the
canonical idiomatic path for entities that are optional/host-configurable.

The entity spec is exported as `adminapi.PluginRemoteSpec()` so the host
application registers it once at startup:

```go
if err := reg.Register(adminapi.PluginRemoteSpec()); err != nil { ... }
```

The spec declares four domain columns (name, url, scope, module — all ColText,
NotNull: true); structural columns (id, tenant_id, version) are injected by fabriq.

## Write path

Writes go through `query.Fabric.Exec` (and `ExecBatch`) which the extension
already holds after `Start` — no extra executor construction needed. For create:

```go
fab.Exec(reqCtx, command.Command{
    Entity:  PluginRemoteEntityType,
    Op:      command.OpCreate,
    AggID:   event.NewID(), // ULID, minted server-side
    Payload: map[string]any{...},
})
```

For delete: `command.OpDelete` with `AggID` set. ID generation uses
`event.NewID()` (ULID via `core/event`), the same function the command executor
uses internally.

## No migration needed

Dynamic entities require no hand-written migration. In production, the adapter
calls `EnsureDynamic` (managed DDL) to create/alter the table on first use. The
`fabriqtest` fakes handle dynamic entities entirely in memory.

## Route shapes

All routes share the admin base path (default `/admin`) and the host-supplied
route options (auth middleware).

```
GET    /admin/plugins         → 200 { "items": [ {id, name, url, scope, module} ] }
POST   /admin/plugins         → 201 { id, name, url, scope, module }
DELETE /admin/plugins/:id     → 204  (404 if not present)
```

Route names: `fabriq.admin.plugins.list`, `fabriq.admin.plugins.create`,
`fabriq.admin.plugins.delete`. Tags: Fabriq, Admin.

## Host registration requirement

Before using plugin endpoints, the host must:

1. Register the entity spec in its registry:
   ```go
   adminapi.PluginRemoteSpec()  // returns registry.EntitySpec
   ```
2. The spec must be registered before `reg.Validate()` is called and before the
   fabriq extension starts (so the real adapter runs EnsureDynamic).
3. The test harness does this via `buildPluginWorld(t)`.
